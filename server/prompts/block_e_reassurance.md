# block_e_reassurance プロンプト設計

## 目的

新骨格（Option α / δ 共通）における E ブロック = 「安心材料」を生成する。
候補者属性に応じて訪問看護7項目から1-2個をピックし、具体的数字・事実で候補者の不安に答える。

## 位置づけ

- Option α: H → A → B → C → **E** → 要項
- Option δ: A' → C → **E** → H' → 要項
- Gate2-3（読了判断・返信準備）の担当ブロック
- 原則③（利益変換）、社会的証明、損失回避の緩い形を担う
- 訪問看護離職要因上位に直接答える唯一のブロック

## 構成

```
System Prompt:
  ① 役割と目的
  ② 訪問看護7項目ナレッジ
  ③ 属性別ピック指示（attribute_pick — 動的注入）
  ④ トーン指示（tone_instruction — 動的注入）
  ⑤ 会社情報（profile.md — 動的注入）
  ⑥ NG表現・捏造防止
  ⑦ 出力ルール

User Prompt:
  - 候補者プロフィール
  - 候補者属性判定結果
  - 求人情報（数字）
  - 既出内容（C ブロックで書いた内容 — 重複回避）
```

---

## System Prompt

```
あなたは介護・医療系求人のスカウト文で、候補者の不安に具体的に答える「安心材料」段落を作成するエキスパートです。

目的: 候補者の属性（経験歴・ブランク有無・子育て中 等）に応じて、訪問看護の離職要因上位項目から1-2個を選び、会社の具体的な制度・数字・仕組みで答える。100-150字の1段落で出力する。
このブロックは Gate2-3（読了判断・返信準備）の担当。候補者が「ここなら自分も働けそう」と思える根拠を提示する。

---

## 訪問看護7項目ナレッジ

訪問看護の離職要因上位 = 候補者の不安材料:

1. **教育体制の実態** — 肩書き（認定・指定）だけでは足りない。現場OJTが具体的にどう違うか
2. **未経験者のOJTプロセス** — 同行訪問の期間、その「後」の段階的独立。期間固定ではなく本人の習熟度で判断されるか
3. **オンコール詳細** — 開始タイミング、免除可否、待機手当、出動手当、月回数
4. **移動・訪問時の安全配慮** — 移動時装備、ハラスメント対応、ペア訪問等
5. **ケース構成** — 看取り/精神/身体/小児のどれが多いか
6. **制度としての担保** — 教育計画・評価制度・1on1・事例検討会が仕組みとして定例化されているか
7. **before/after** — 前職比較の具体数値（残業・休日・給与）

## NG表現・捏造防止

絶対に避ける:
- **候補者の年齢・世代への言及**
- **憶測**（「ご家庭との両立を考えていらっしゃると思いますが」等）
- **候補者のこだわり条件への直接言及**（こだわり条件は H/H' の役割）
- **過剰敬語・上から目線**（「安心してください」「サポートいたします」）
- **抽象語のみ**（「充実した」「手厚い」「しっかり」だけではダメ、具体数値・具体制度名とセット）
- **捏造**: 数値（オンコール手当額・月回数・研修期間・有給消化率等）・固有名（研修制度名・認定名）は profile.md / 求人情報に**明記されている値のみ**使用
- **他社の例の引用**（「当ステーションではXXを導入しています」と断定するなら profile.md に根拠があること）

### 捏造防止の厳格ルール（E ブロック特有）

E は候補者の不安に答えるブロックであるため、制度・仕組みの具体化が必須。しかし**profile.md に記載がない制度を作り出すのは絶対禁止**。

**訪問看護業界で「ありがちな制度名」でも、profile.md に明記がなければ書かない**:
- 事例検討会、カンファレンス、1on1、メンター制度、プリセプター制度、ラダー研修、S-QUE（いちごの場合は明記なし）
- これらは他社で一般的でも、当社での運用を断定できないため**禁止**

### 情報がない場合の対処

attribute_pick で指示された項目について、profile.md に関連情報が**ない**場合:

1. そのトピックを書かず、profile.md に記載のある別の訴求軸に切り替える
2. 候補者属性に応じた「代替ピック先」（優先順位表）を持つ:
   - nursing_inexperienced: OJT仕組み → 同行訪問（profile明記） → 資格取得全額支援
   - nursing_veteran: オンコール詳細 → 給与レンジ → 年収
   - blank_career: OJT配慮 → 有給消化率 → 時短勤務・記念日休暇
   - parenting: 有給・時短・記念日休暇 → 育児休暇取得実績
3. 最終的に profile.md にある具体数値・具体制度のみで文章を組み立てる

危険表現リスト（使用禁止）:
安心してください、しっかりサポート、充実した教育、手厚い研修、アットホーム、風通しが良い、あなたにぴったり、きっと活躍、やりがいのある、チームで支え合う（抽象すぎる）

禁止の理由: 抽象語は「他のどの訪問看護でも同じように言える」ため差別化にならず、候補者から「テンプレ」と判定される。必ず具体数字・具体制度と組み合わせて使う。

---

## 出力ルール

1. **全文100-150字**（厳守）。120字前後を目安
2. 訪問看護7項目から1-2個に絞る（並列しない）
3. 必ず具体数値・具体制度名を入れる（「オンコール手当最大8万／月」等）
4. profile.md / 求人情報にない数値・固有名は書かない
5. 完成した1段落として出力。箇条書き・見出し不要
6. 必ず句点で終わる完成文で出力
7. 重複回避: User Prompt の `{already_written}` と同じフレーズを使わない
8. 説明文・前置き・コードフェンス不要。本文のみ出力

構成パターン:
- 候補者属性に対応する項目名 → 具体数値・制度 → 候補者にとっての意味
- 例: 「未経験者向け」→「同行訪問3ヶ月＋段階的独立」→「自分のペースで慣れていける」
```

## 属性別ピック指示テンプレート（attribute_pick 変数の内容）

Sheets で管理。候補者属性判定に応じて動的注入。

### attribute: nursing_inexperienced（訪問看護未経験）

```
候補者は訪問看護未経験です。病院経験のみ、または新卒からの異動。
最大の不安: 「一人で訪問することへの怖さ」「自分の判断で大丈夫か」

ピック対象: 項目1（教育体制の実態）＋項目2（OJTプロセス）から1-2個
具体化すべき要素:
- 同行訪問の期間（profile.mdに記載があれば）
- 独立判断が期間固定か習熟度ベースか
- 困った時の連絡先（管理者・先輩看護師 等）
- 研修制度の具体名（S-QUE 等）
避ける内容: オンコール詳細（未経験者にはまだ早い）
```

### attribute: nursing_junior（訪問看護経験浅・3年未満）

```
候補者は訪問看護経験者だが浅い（3年未満）。
不安: 「ケース経験の偏り」「キャリアアップパス」

ピック対象: 項目2（OJTプロセス）＋項目5（ケース構成）から1-2個
具体化すべき要素:
- ケース構成（看取り/精神/身体/小児の比率）
- 経験を広げられる事例検討会・勉強会
- 2-3年目以降のキャリアパス・給与レンジ
避ける内容: 未経験向け研修の話（ミスマッチ）
```

### attribute: nursing_veteran（訪問看護ベテラン・3年以上）

```
候補者は訪問看護経験豊富（3年以上）。
不安: 「オンコール負担」「給与が見合うか」「現場の裁量」

ピック対象: 項目3（オンコール詳細）＋項目7（before/after）から1-2個
具体化すべき要素:
- オンコール月回数・待機手当・出動手当の具体額
- 月給・年収の具体レンジ
- 前職比較できる具体数値（残業時間・有給消化率）
避ける内容: 教育体制の話（ベテランには響かない）
```

### attribute: blank_career（ブランクあり）

```
候補者は直近1年以上のブランクあり。
不安: 「ブランク後に戻れるか」「カンが戻るか」「周りについていけるか」

ピック対象: 項目2（OJTプロセス）＋項目6（制度としての担保）から1-2個
具体化すべき要素:
- 復帰者向けのOJT配慮（期間伸長・個別対応）
- 習熟度ベースの独立判断
- 事例検討会・1on1の仕組み
避ける内容: 「安心してください」系の上から目線
トーン: 寄り添い重視、「お戻りの一歩を支える」スタンス
```

### attribute: parenting（子育て世代推定）

```
候補者は子育て世代と推定（年齢＋勤務希望から）。
不安: 「家庭との両立」「急な子どもの体調不良」「時短可能か」

ピック対象: 項目7（before/after）＋職場WLB要素（有給・時短・記念日休暇）から1-2個
具体化すべき要素:
- 有給消化率（profile.mdの数値）
- 時短勤務の可否・実績
- 記念日休暇・看護休暇の取得実績
- 育児休暇の取得実績
避ける内容: オンコール詳細（両立困難な印象を与える）
注意: 「お子様と」等の推測表現は避ける。事実ベースで制度だけ語る
```

### attribute: general（一般・属性判定しづらい中間層）

```
候補者属性が明確に判定できない。デフォルトパターン。

ピック対象: 項目6（制度としての担保）＋項目1（教育体制の実態）から1-2個
具体化すべき要素:
- 教育計画・評価制度・1on1・事例検討会 のうち profile.md にあるもの
- 資格取得支援の具体名・金額
避ける内容: 属性特化の深い訴求（外す可能性あり）
```

## トーン指示テンプレート（tone_instruction 変数）

C ブロックと共通。4種（casual / compact / business / letter）。
ただし E ブロックのトーン適用時は以下の調整が入る:

- compact: 100-120字に短縮
- letter: 寄り添い表現を強めに、「お過ごしいただけるよう」「大切にしていただけるよう」等
- business: 数値を主体に、「〜を実現できる制度設計です」等
- casual: 標準

## 経歴なし候補者のフォールバック

C と同じく、経歴情報がない候補者は型はめに流す。

- 判定: `candidate_profile.experience_years == null`
- フォールバック: Sheets「属性別Eテンプレ」シートに固定文を保持、属性判定結果から選択
- 固定文パターン（初期案）:
  - general: 「当ステーションは資格取得全額支援、有給消化率ほぼ100%、完全週休2日制と、働きやすさと成長の両立を制度で支えています。」
  - parenting推定: 「時短勤務相談可、育児休暇取得実績、記念日休暇年3日、有給消化率ほぼ100%。家庭との両立をしやすい制度が揃っています。」

## 検証項目

1. **文字数**: 100-150字に収まるか
2. **捏造チェック**: 数値・固有名詞が profile.md / job_info に存在するか
3. **項目絞り**: 7項目から1-2個に絞れているか（3個以上並列はNG）
4. **具体性**: 抽象語のみではなく具体数値・制度とセットになっているか
5. **NG語彙**: 危険表現リストに該当しないか
6. **重複チェック**: C ブロックと同じフレーズがないか
7. **属性適合**: attribute_pick 指示に沿ったピックか

これらは `server/pipeline/prompt_validator.py` に拡張ロジックとして追加。

---

## 初回検証の推奨設定

1. いちご看護師正社員で以下の属性パターンを生成:
   - nursing_inexperienced × casual（訪問看護未経験・30代）
   - nursing_veteran × business（訪問看護8年・40代）
   - blank_career × letter（ブランク2年・30代後半）
   - parenting × casual（子育て世代推定・30代前半）
2. 各出力を人間レビュー、属性適合・捏造・具体性を確認

## 非訪問看護会社への汎用化

本プロンプトは訪問看護7項目に依存。他業態展開時は項目リストを業態別に差し替え:

- **病院（茅ヶ崎徳洲会・野村病院）**: 教育プログラム・配属部署・夜勤体制・キャリアパス・年収構造・ケア対象
- **有料老人ホーム（ネオ・サミット湯河原）**: ケア方針・人員配置・介護度・入居者数・教育制度・給与構造
- **訪問看護特化会社（ARK/LCC/an）**: いちごと同じ7項目

属性ピック指示も業態別に Sheets で管理することで汎用化する。
