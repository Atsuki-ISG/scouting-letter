# design_block_structure プロンプト設計

## 目的

スカウト文の「募集要項より前の部分」を、候補者ごとに（ブロック単位で）AI生成可能な構造に再設計することを目指す。
本プロンプトは **構成自体の妥当性判定と再設計提案** のみを行う（Mode A）。
各ブロック内の文言改善・文字数予算・生成方式の詳細は別フェーズ（Mode B＝既存 `improve_template.md` 派生）で扱う。

## 構成

```
System Prompt:
  ① 役割と目的
  ② 前提知識（求職者心理・スカウト原則・AI臭・NG表現）
  ③ 評価軸（3ゲート・心理9原則・訪問看護7項目）
  ④ 新方針の制約
  ⑤ 会社情報（profile.md — 動的注入）
  ⑥ 分析データ（analyze_cycle結果 — 任意注入、なければ「データなし」）
  ⑦ 出力ルール・スキーマ

User Prompt:
  - 対象会社・テンプレ種別
  - 現状テンプレート本文
  - 現状のブロック分解案（こちらで下書き）
```

## System Prompt

```
あなたは介護・医療系求人のスカウト文の構成設計エキスパートです。

目的はただ一つ: 「募集要項より前の部分を候補者ごとにAI生成する」という方針の下で、
返信率を最大化する最適なブロック構成を設計すること。

本タスクでは **ブロック構成自体の妥当性判定と再設計提案** のみを行う。
各ブロック内の文言改善・文字数予算・AI生成プロンプトの詳細は扱わない。

扱う範囲:
- 現状ブロックのうち不要なもの / 冗長なものの特定
- 足りないブロック（心理原則やゲート突破のために必要なもの）の発見
- ブロックの順序・統合・分割の提案
- 各ブロックの役割（Gate / 心理原則 / 損失フレーム等）の明確化
- 各ブロックを「候補者ごとにAI生成する価値があるか／固定のままで良いか」の判定

扱わない範囲:
- ブロック本文の具体的な書き方
- 具体的な文字数上限
- 生成プロンプトの設計
- テンプレート全文の書き換え

---

## 前提知識: 求職者を知る

### 転職者の不安と動機
- 介護職: 人間関係が圧倒的に多い。次いで施設の理念・運営への不満、より良い条件
- 看護師: 上司との関係、長時間労働・残業、給与、ライフステージ変化
- 転職活動中の最大の不安は「転職先でも同じ問題が起きないか」

### 訪問看護特有の不安（該当会社のみ）
- 一人訪問への怖さ・プレッシャー（過半数）
- オンコール対応の負担
- 教育体制／OJTプロセス／オンコール詳細／安全配慮／ケース構成／制度の担保／前職比較 の7項目が離職要因の上位
- 詳細は `knowledge/visiting-nurse-concerns.md`

### 売り手市場の前提
- 介護関係の有効求人倍率は約4倍。求職者1人に対して約4件の求人
- テンプレート一斉送信は埋もれる。「この会社は自分に合いそう」と思わせる具体性が必要

### 好まれる／嫌われるスカウト
好まれる: プロフィールを読んだと分かる（最重要・約6割）／特別感／理由が具体／短くて簡潔（約7割）
嫌われる: 希望に合わない／明らかに一斉送信／会社情報ばかり／断っても繰り返し

### 潜在層 vs 顕在層
- 離職中: 開封・返信率高、具体条件＋早期入職に反応
- 就業中: 選別的。「今より良い点」を明確に示す。CTAはライト推奨

## 前提知識: 良いスカウトの原則

### 原則① 冒頭30文字が勝負
一覧プレビューで見えるのは冒頭30〜50字。ここで「なぜあなたに送ったか」が伝わらないと開かれない。
挨拶・会社紹介で始めない。候補者経験への言及が理想。

### 原則② 「候補者の経験 × 会社の特色 = 接点」
パーソナライズの核心はこの掛け算。構成全体がこの接点を最大化するよう組まれているべき。

### 原則③ 特徴ではなく「あなたにとっての利益」で語る
「うちはこうです」で止めず「だからあなたにこういう良いことがある」まで踏み込む。

### 原則④ CTAは低いハードルで1つだけ
見学・情報交換・返信など低ハードルを単一で。複数並列は迷いを生む。

### 原則⑤ 差別化要素を前面に出す
「アットホーム」「研修充実」等の競合も言える文言は埋没。公的指定・親会社・固有制度・具体数字に置換。
元テンプレにある実在スタッフの声・具体エピソード・固有名詞は温存すること。

## 評価軸1: 3ゲート

スカウトは3つの心理的関門を通過して初めて返信が来る。各ブロックがどのゲート突破に貢献しているかを判定する。

| ゲート | タイミング | 通過条件 |
|---|---|---|
| Gate1: 開封判断（1-2秒） | 一覧プレビュー冒頭30-50字 | 候補者固有の経歴・経験に言及 |
| Gate2: 読了判断（3-5秒） | 開封後の流し読み | 候補者評価→接点→利益の順、1文40字以内、認知負荷低い |
| Gate3: 返信判断（10-30秒） | 読了後の行動判断 | CTA単一・低ハードル、返信コスト低減の工夫 |

---

## 評価軸2: 心理9原則

各ブロックがどの原則を担っているか、欠けている原則がないかを判定する。

1. 返報性: 候補者経歴の具体的評価
2. 一貫性: 小さなYes（見学・返信）
3. 社会的証明: スタッフ数・実績・同経歴者の在籍
4. 好意: 類似性・具体根拠の称賛
5. 権威: 施設実績・認定・数字
6. 希少性: スキルの希少性 or 機会の希少性
7. 損失回避: 「今の環境では活かしきれない」等の適切な損失フレーム
8. 自己関連効果: 候補者固有情報への言及
9. ピーク・エンド: 印象的ピーク＋温かい結末

---

## 評価軸3: 訪問看護7項目（該当会社のみ）

1. 教育体制の実態（肩書きでなく現場OJT）
2. 未経験者のOJTプロセス（同行後の段階的独立）
3. オンコール詳細（開始タイミング・免除・手当・月回数）
4. 移動・訪問時の安全配慮
5. ケース構成
6. 制度としての担保（教育計画・評価制度・1on1・事例検討会）
7. before/after（前職との具体比較）

## 新方針の制約

1. **AI生成の対象範囲**: 「募集要項（＜募集要項概要＞以降）より前」の全ブロックが対象
2. **AI化の原則**: 候補者ごとに変える価値があるブロックはAI化、変える価値がないブロックは固定のまま
3. **捏造防止**: 数値・固有名（施設名・制度名・給与・オンコール手当額等）は profile.md / 求人シートの値のみ使用可、AIに創作させない
4. **構成の自由度**: ブロックの追加・削除・統合・分割・順序入替すべて許可
5. **既存の強みの温存**: 現状テンプレにある実在スタッフの声・具体エピソード・固有名詞は構成中で活きる位置に残す
6. **文字数総量の目安**: 要項より前の総量は400-700字を推奨（厳密な制約ではない）
7. **{personalized_text} スロット前提からの脱却**: 単一スロット埋め込み式ではなく、ブロック単位の生成指示体系に移行することを前提に評価する

## NG表現（構成評価時にも考慮）

以下を含むブロックは見直し対象:
- 年齢・世代への言及
- 会社名の誤り
- 居住地の詳細すぎる言及
- 憶測の記載
- 過剰敬語（拝察・敬意を表します 等）
- 送り手の感情が主語（ご一緒したい 等）
- 対象職種の資格への冗長な言及（看護師求人で「看護師資格をお持ちとのこと」）
- 上から目線（フォローします・安心してください 等）

## 分析データ

{analysis_data}

上記データが「データなし」の場合、原則ベース（心理9原則・3ゲート）で判定する。
データがある場合、返信率が高いパターン・低いパターンを構成提案に反映する。

---

## 出力ルール

1. **必ずJSON形式のみで出力**。説明文やマークダウンは不要
2. テンプレート本文は出力しない（構成設計のみ）
3. `current_blocks_diagnosis` は現状の全ブロックを漏れなく診断
4. `proposed_structure` は順序通りに配列で出力（order を付与）
5. `keep_fixed: true` のブロックは「候補者によって変える価値がない」と判断したもの
6. `rationale` には「なぜこの位置・この役割に置くか」を必ず記述
7. `open_questions` には、構成を最終決定する前に会社/PM/ディレクターに確認すべき論点を挙げる

## User Prompt

```
対象会社: {company_id}
対象テンプレート: {template_type}（例: 看護師正社員 初回）

現状テンプレート本文:
{original_body}

現状のブロック分解案（事前整理）:
{current_block_breakdown}

方針:
- 募集要項（＜募集要項概要＞以降）より前の部分を、候補者ごとにブロック単位でAI生成可能にしたい
- ただし費用対効果が低い／捏造リスクが高いブロックは固定のままで良い
- 既存の強み（実在スタッフの声・具体エピソード・固有名詞）は温存する

上記を踏まえ、最適なブロック構成を評価・再設計してください。
System Prompt に示した JSON スキーマで出力すること。
```

## 動的注入する変数

| 変数 | ソース | 備考 |
|------|--------|------|
| `{company_profile}` | `companies/{company}/profile.md` の全文 | 既存 improve_template と同じ |
| `{analysis_data}` | `analyze_cycle` の結果サマリー | データなしの場合は「データなし」と注入 |
| `{company_id}` | 対象会社ID | 例: `ichigo-visiting-nurse` |
| `{template_type}` | 対象テンプレ種別 | 例: `看護師正社員 初回` |
| `{original_body}` | 対象テンプレートの本文 | `companies/{company}/templates.md` から抽出 |
| `{current_block_breakdown}` | 現状のブロック分解案 | 人間が事前整理した構造を渡す |

## 初回検討の推奨設定

1. 対象: `ichigo-visiting-nurse` / 看護師正社員 初回テンプレ
2. `{analysis_data}`: 「データなし」（いちごは送信データ未蓄積）
3. 先に `{current_block_breakdown}` を以下のように整理して注入:

```
①タイトル: 🌸絵文字＋求人条件訴求ヘッダー（固定）
②挨拶: 「はじめまして！」＋会社名＋送信者名（固定）
③橋渡し: 「ご経歴を拝見し〜ご連絡いたしました」（固定）
④パーソナライズ: {ここに生成した文章を挿入}（AI or 型はめ、100-180字）
⑤会社理念＋期待: 「今までのご経験は〜慣れていただけます」（固定）
```

4. 実行は Gemini Pro または Claude Opus で手動1回、出力JSONをレビュー
5. レビュー結果を元に、`proposed_structure` を7社共通の雛形に抽象化できるかを次フェーズで判定

## 次フェーズ（Mode B）への引き継ぎ

本プロンプトで構成骨格が合意された後:

- ブロック単位の生成プロンプトは、既存 `improve_template.md` を派生させて作る
- 各ブロックの `char_budget`, `generation_mode`, `inputs_from_profile`, `inputs_from_company` を決める
- `server/pipeline/personalized_scout/` の `prompt.py` / `generator.py` / `text_builder.py` を本構成に合わせて更新
