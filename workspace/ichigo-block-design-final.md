# いちご スカウト文ブロック構成 最終骨格（Option α / v1.2-final）

**確定日**: 2026-04-23
**対象**: ichigo-visiting-nurse / 看護師正社員 初回テンプレート
**スコープ**: 募集要項（＜募集要項概要＞以降）より前の全ブロック

---

## 決定事項

Option α（H-A-B-C-E の5ブロック）を採用。**Option δ と並行実装し、候補者属性で使い分ける二本立て運用**とする（2026-04-23 追記）。

**トーンは骨格とは独立したパラメータ**（2026-04-23 追記）。ブロック構造はトーンで変わらず、各ブロック内の文体・敬語レベル・装飾を `{tone_instruction}` 変数で制御する。対応トーン4種:

| トーン | 特徴 | 想定候補者 |
|---|---|---|
| カジュアル（現状） | 「はじめまして！」フランク敬語、ほどよい軽さ | 一般・若手〜中堅 |
| コンパクト | 短文・装飾少、ブロック内字数を30%程度圧縮 | 高年収志向・忙しい層 |
| ビジネス | 「です/ます」中心、感嘆符なし、数字事実ベース | 管理職・大手経験者 |
| お手紙 | 寄り添い・気遣い・温度感、改行多め | ブランク・育児中・共感求める層 |

（骨格, トーン）のペアを候補者属性で選択。実装はプロンプトの `{tone_instruction}` 変数＋Hプールのトーン別テキストバリエーションで対応。

**受容したトレードオフ**:
- モバイル50字プレビュー枠でのGate1b（自己関連効果）突破が弱い（Hが46字でプレビュー枠をほぼ消費、Aは見えない確率が高い）
- Gate1a（条件訴求）はHが担うため、モバイルでもGate1自体は突破可能
- PC 90字プレビュー枠では H＋B＋A の冒頭まで見え、Gate1a＋Gate1bの両立が成立

**選択理由（推定）**:
- 現状🌸ヘッダー訴求の維持がクライアント/運用上重要
- ヘッダー廃止（Option δ）は会社の顔を失うリスクが大きい
- PC開封者にはGate1b機能、モバイル開封者は条件訴求Gate1aで受け止める二段構え

## 確定骨格

```
┌──────────────────────────────────────────────────┐
│ H ヘッダー行（求人条件訴求）                      │ 固定 約46-60字
│   🌸絵文字＋拠点特徴＋差別化数字1-2軸             │ 強調軸選択はAI or 型はめ
├──────────────────────────────────────────────────┤
│ A 冒頭接点（候補者固有×会社特色の1文）            │ AI生成 40-60字
│   経歴なし候補者はスキップ                        │ Gate1b担当
├──────────────────────────────────────────────────┤
│ B 挨拶・送信者名                                  │ 固定 約30-40字
│   「はじめまして、いちごの柳田です。」レベル     │ 目的宣言は削除
├──────────────────────────────────────────────────┤
│ C パーソナライズ本体                              │ AI生成 150-200字
│   候補者評価＋接点深掘り＋利益変換を1段落で統合   │ 経歴なしは型はめ（recipes.md A-G）
├──────────────────────────────────────────────────┤
│ E 安心材料（候補者属性別にピック）                │ AI選択＋固定 100-150字
│   教育/OJT/オンコール/ケース構成/WLB から1-2個    │ 属性ベースで選択
├──────────────────────────────────────────────────┤
│ ＜募集要項概要＞以降はスコープ外（固定運用継続）  │
└──────────────────────────────────────────────────┘

全体文字数目安: 約400-520字（現状約480字）
```

## 残 open_questions とデフォルト案

v1.1 レビューで残った論点について、PM/ディレクター判断を得るまでのデフォルト案：

### Q1: プレビュー表示文字数の実機確認（済）

**確認結果**:
- モバイル: 約50字（2行でtruncate）
- PC: 約90字（3行でtruncate）
- 事業所名はジョブメドレー自動表示（テンプレ制御外）

**影響**: Option α はモバイルでGate1bが弱い。受容済み。

### Q2: Hの訴求軸切替パターン数 — **プール＋こだわり条件マッピング方式に変更**（2026-04-23追記）

Chrome拡張で既にこだわり条件を抽出済み（`extension/src/content/selectors.ts:76` の `specialConditions`）。これを使って H を完全型はめ運用する。

**プール設計（8個案）**:

| プールID | こだわり条件 | H訴求案 |
|---|---|---|
| P01 | 高収入／年収アップ | 🌸【年収500-600万可】月給35-45万／オンコール手当月最大8万 |
| P02 | （デフォルト／該当なし） | 🌸【富士見台5月オープン】立ち上げメンバー募集／年収500-600万可 |
| P03 | ブランク可／未経験可／教育体制 | 🌸訪問看護未経験歓迎／資格取得全額支援／同行訪問＋研修充実 |
| P04 | 残業少／土日休み／WLB | 🌸完全週休2日（土日祝）／年間休日120日以上／有給消化率ほぼ100% |
| P05 | 産休育休／託児所／時短 | 🌸時短勤務相談可／記念日休暇年3日／育児休暇取得実績あり |
| P06 | 資格取得支援／スキルアップ | 🌸資格取得全額支援（セミナー・受験料・書籍購入）／昇給年6万 |
| P07 | 車通勤可／駅近 | 🌸電動アシスト自転車・バイク・車貸与／交通費支給／副業OK |
| P08 | 残業少／定時帰り（P04と近いが短時間重視） | 🌸残業ほぼなし／9:00-18:00定時／昇給年6万／有給ほぼ100% |

**マッピングルール**:
- 候補者のこだわり条件を読み → マッピング表で対応プールを決定
- 複数該当時は候補者の優先順位が取れれば上位、取れなければマッピング表上位を採用
- こだわり条件なし or 該当なし → P02 デフォルト（新規オープン訴求）

**実装方針**:
- プールとマッピングは Google Sheets で管理（非エンジニア編集可）
- AI生成ゼロ、完全型はめ運用（捏造リスク最小）
- orchestrator で `こだわり条件 → プールID → テキスト` の3段階処理

**既存ルールとの整合**:
- `companies/ichigo-visiting-nurse/templates.md:20` の「10. こだわり条件への言及は控える」はパーソナライズ文（C）内での直接言及禁止を意味する
- Hでの訴求軸マッチングは候補者への直接言及ではなく訴求の選択なので整合OK
- C ブロックのプロンプト設計時に、こだわり条件への言及禁止を明示する

## 運用: α/δ の使い分け（属性別二本立て）

Option δ と並行実装し、**候補者属性で骨格を使い分ける**。

### 振り分け仮説マトリクス（骨格 × トーン）

| 候補者属性 | 骨格 | トーン | 理由 |
|---|---|---|---|
| 経歴なし・情報少 | α | カジュアル | A スキップ→型はめが自然 |
| ベテラン・条件重視（経験5年以上・40代以上） | α | ビジネス or コンパクト | 条件訴求層、感情より事実重視 |
| 高年収志向（こだわり＝高収入） | α | コンパクト | 要点即判定で P01 プールを最大効果化 |
| 管理職経験・大手出身 | α | ビジネス | 敬語・事実ベースが相性良い |
| 中堅・若手（20-30代前半） | δ | カジュアル | モバイル50字でGate1bを通す |
| 訪問看護未経験・病院出身 | δ | カジュアル | 個別対応感を前面に |
| ブランクあり | δ | お手紙 | 温度感・寄り添いが重要 |
| 育児中・共感求める層 | δ | お手紙 | 気遣い表現で刺さる |
| 自己PR豊富・経歴情報豊富 | δ | カジュアル or お手紙 | A' に情報展開する余裕 |
| 中間層 | α | カジュアル（デフォルト） | クライアント訴求維持 |

- 振り分けロジックは Google Sheets で管理（非エンジニア編集可）
- 初期は仮説ルール、運用データで調整

### 検証方法

- ❌ 旧: ランダム50/50 で α vs δ 勝ち残り
- ✅ 新: 属性別に最適骨格を割当、返信率を α内 × δ内 それぞれで追跡
- A/B比較は特定属性セグメント内で並走（例: 中堅層で α/δ ランダム振り分け）

### 仮説検証の優先度（自信度低い順）

1. モバイル開封比率が年代で本当に違うか — 実データで確認必要（最優先）
2. 「ベテラン＝条件訴求／若手＝個別対応」の区分妥当性
3. 「訪問看護未経験＝δ」の有効性
