# いちご スカウト文ブロック構成 最終骨格（Option δ / v1.2-final-delta）

**確定日**: 2026-04-23
**対象**: ichigo-visiting-nurse / 看護師正社員 初回テンプレート
**スコープ**: 募集要項（＜募集要項概要＞以降）より前の全ブロック
**運用形態**: Option α と並行運用（A/B比較）

---

## 決定事項

Option δ（ヘッダー廃止＋本文冒頭A融合 の4ブロック）を Option α と並行で実装し、返信率をA/B比較する。

**選択理由**:
- モバイル50字プレビュー枠で Gate1b を確実に突破できる唯一の構造
- エール訪問看護など他社で実績あるパターン
- 差別化訴求を要項直前に分離することで捏造リスク最小
- Option α がモバイルで Gate1b 弱いトレードオフを抱えるため、実データで検証する価値あり

---

## 確定骨格

```
┌──────────────────────────────────────────────────┐
│ （事業所名）いちご訪問看護リハビリステーション     │ ジョブメドレー自動表示
│   — Gate1a はここで担保                           │
├──────────────────────────────────────────────────┤
│ A' 挨拶＋候補者接点の融合ブロック                  │ AI生成 80-120字
│   「はじめまして、いちごの柳田です。              │ Gate1a補完＋Gate1b担当
│    [候補者固有×会社特色の1文]」                  │ 経歴なしはスキップ→B'で代替
├──────────────────────────────────────────────────┤
│ C パーソナライズ本体                              │ AI生成 150-200字
│   候補者評価＋接点深掘り＋利益変換を1段落で統合   │ 経歴なしは型はめ（recipes.md A-G）
├──────────────────────────────────────────────────┤
│ E 安心材料（候補者属性別にピック）                │ AI選択＋固定 100-150字
│   教育/OJT/オンコール/ケース構成/WLB から1-2個    │ Option α と共通
├──────────────────────────────────────────────────┤
│ H' 求人ハイライト行                               │ 固定 約40-50字
│   🌸絵文字＋拠点特徴＋差別化数字1-2軸             │ 要項直前に配置
├──────────────────────────────────────────────────┤
│ ＜募集要項概要＞以降はスコープ外（固定運用継続）  │
└──────────────────────────────────────────────────┘

全体文字数目安: 約380-520字（Option α とほぼ同等）
ブロック数: 4（Option α の5より1少ない）
```

## 候補者タイプ別の生成パス

| 候補者タイプ | A' | C | E | H' |
|---|---|---|---|---|
| 経歴あり（情報豊富） | AI生成（挨拶＋経歴接点1文） | AI生成 | AI選択 | 固定＋軸選択 |
| 経歴あり（情報少） | AI生成（挨拶＋短い接点） | AI生成 or 型はめ | AI選択 | 固定＋軸選択 |
| 経歴なし（就業中推定） | **挨拶部分のみ**（接点1文はスキップ） | 型はめ（recipes.md） | 型はめ連動 | 固定＋軸選択 |
| 経歴なし（未就業） | **挨拶部分のみ** | 型はめ | 型はめ連動 | 固定＋軸選択 |

**A'の構造**:
- 経歴あり: 「はじめまして、[会社名＋送信者名]です。[候補者固有×会社特色の接点1文]」
- 経歴なし: 「はじめまして、[会社名＋送信者名]です。」のみ（短縮）

## Option α との共通化可能領域

| ブロック | α | δ | 共通化 |
|---|---|---|---|
| C パーソナライズ本体 | ○ | ○ | **完全共通化可能** |
| E 安心材料 | ○ | ○ | **完全共通化可能** |
| 挨拶部分 | B（独立） | A'の前半 | プロンプト分離必要 |

**Mode B 実装時の方針**:
- C・E のプロンプトは α/δ 共通で1本作る
- A (α) と A' (δ) は別プロンプトで2本作る（長さ・挨拶融合の有無が異なる）
- H と H' はテキスト生成ロジック共通、配置位置のみ組み立て時に切替

## 運用形態：候補者属性別の使い分け（2026-04-23 方針変更）

α/δ は「勝ち残り1本」を決めるのではなく、**候補者属性によって使い分ける二本立て運用**とする。

### 振り分け仮説マトリクス（骨格 × トーン）

トーンは骨格と独立したパラメータとして扱う（ブロック構造はトーンで変わらず、ブロック内の文体のみ変化）。

| 候補者属性 | 骨格 | トーン |
|---|---|---|
| 経歴なし・情報少 | α | カジュアル |
| ベテラン・条件重視（経験5年以上・40代以上） | α | ビジネス or コンパクト |
| 高年収志向（こだわり＝高収入） | α | コンパクト |
| 管理職経験・大手出身 | α | ビジネス |
| 中堅・若手（20-30代前半） | δ | カジュアル |
| 訪問看護未経験・病院出身 | δ | カジュアル |
| ブランクあり | δ | お手紙 |
| 育児中・共感求める層 | δ | お手紙 |
| 自己PR豊富・経歴情報豊富 | δ | カジュアル or お手紙 |
| 中間層 | α | カジュアル（デフォルト） |

### 実装方式

- 振り分けロジックは Google Sheets で管理（非エンジニア編集可）
- 初期は仮説ルール、運用データで調整
- orchestrator で候補者属性 → 骨格選択 → 対応パイプライン呼び分け

### 検証方法の変更

- ❌ 旧: ランダム50/50 で α vs δ 勝ち残り決定
- ✅ 新: 属性別に最適骨格を割当、返信率を α内 × δ内 それぞれで追跡
- A/B比較は**特定属性セグメント内で並走**（例: 中堅層でα/δランダム振り分け）

### 仮説検証の優先度（自信度低い順）

1. モバイル開封比率が年代で本当に違うか — 実データで確認必要（最優先）
2. 「ベテラン＝条件訴求／若手＝個別対応」区分の妥当性 — 運用2-4週後の返信率クロス集計で確認
3. 「訪問看護未経験＝δ」の有効性 — 同上

### 検証期間・指標

- 最低2週間・各バージョン100件以上送信
- 主要指標: 返信率（version × 候補者属性でクロス集計）
- 副次指標: 開封率（取得可能なら）、テンプレ別返信率
- 終了条件: 統計有意差が出たら勝ち残り、出なければ他会社展開で再検証
