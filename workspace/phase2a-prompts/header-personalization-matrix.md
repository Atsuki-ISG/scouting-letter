# ヘッダーパーソナライゼーション設計（Phase 2-C）

**方針**: ジョブメドレー標準の希望条件タグを共通語彙とし、候補者の `special_conditions` と会社の売り文句を intersection でマッチングして、候補者ごとにヘッダーをパーソナライズする。

**データソース**: `workspace/competitor-research/2026-04-18_*.md`（7社分）＋ `companies/*/profile.md` から事実ベースで抽出。捏造しない（[feedback_no_fabrication](memory)）。

## 0. preview 最適化原則（2026-04-21 JM UI観察から確定）

ジョブメドレーのスカウト一覧 preview で既に別表示される情報は、ヘッダー本文で**繰り返すと無駄**になる:

| 項目 | preview 表示位置 | → ヘッダー本文で繰り返すか |
|---|---|---|
| 施設名・会社名 | タイトル（teal文字） | **書かない** |
| 画像バナー | 左サムネイル | **書かない（画像別管理）** |
| 職種 | タグ（看護師/准看護師） | **書かない** |
| 地域（都道府県＋区市） | タグ（東京都荒川区） | **より具体的に書く**（町屋駅徒歩5分等） |
| 雇用形態 | タグ（正職員） | **書かない** |
| 給与レンジ | 独立表示（月給325,000円〜450,000円） | **書かない**（ただし"インセンティブあり"等補足はOK） |
| preview本文 | 冒頭 **~80字**（モバイル）〜100字（デスクトップ） | **ここに差別化要素を凝縮** |

### 優秀な preview 例（観察実物）

> 【荒川区（町屋駅徒歩5分）｜直行直帰も可】月35万＋インセンティブで頑張りを還元！終末期ケア…

**なぜ強いか**:
- 【】で具体アクセス＋働き方を凝縮
- 月35万の"+インセンティブ"で給与レンジ+α情報
- 終末期ケアで職種内の専門訴求

### 弱い preview 例

> はじめまして。プロフィールを拝見して、ぜひ直接会ってお話したいと思いスカウトを…

**問題**: 全社が同じ文言で書く → 差別化ゼロ。冒頭80字を挨拶で潰すのは大損。

### ヘッダーに入れるべき3セグメント（優先度順）

1. **具体的アクセス**（〜20字）: 「○○区｜○○駅徒歩○分」「○○市内／直行直帰可」
2. **独自差別化**（〜30-40字）: 独自福利・希少性・独自強み（競合で未確認のもの優先）
3. **働き方・候補者属性マッチ**（〜30字）: 候補者 special_conditions とマッチした売り文句

## 2. Sheets データ構造（売り文句プール）

新規シート: `売り文句`

| カラム | 説明 |
|---|---|
| `company` | 会社ID（ark-visiting-nurse 等） |
| `id` | 売り文句ID（sp01, sp02...） |
| `content` | ヘッダーに入る文字列（例: "年間休日120日以上"） |
| `category` | カテゴリ（休日 / 勤務時間 / 給与 等） |
| `jm_tags` | JM希望条件タグ（カンマ区切り、候補者マッチング用） |
| `source` | 出典（profile.md §おすすめ(2) / 求人シート row3 / 競合調査 §強み① 等）**必須** |
| `priority_base` | 会社としての押し推し度（1-5） |
| `target_job` | 対象職種（看護師 / PT / OT / 全職種） |
| `target_employment` | 対象雇用形態（正社員 / パート / 全て） |
| `exclusive_with` | 排他ルール（例: sp02="オンコールなし" と sp03="オンコール手当充実" は両立不可） |

## 3. スコアリングロジック

```python
def rank_selling_points(candidate_special_conditions, selling_points, job, employment):
    scored = []
    for sp in selling_points:
        # 職種・雇用形態でフィルタ
        if sp.target_job not in [job, "全職種"]: continue
        if sp.target_employment not in [employment, "全て"]: continue
        
        # intersection count
        match_count = len(set(candidate_special_conditions) & set(sp.jm_tags))
        score = sp.priority_base + 2 * match_count
        scored.append((sp, score, match_count))
    
    # 排他ルール適用（上位スコアから選択、排他グループを蹴る）
    selected = []
    seen_exclusives = set()
    for sp, score, _ in sorted(scored, key=lambda x: -x[1]):
        if sp.exclusive_with and any(e in seen_exclusives for e in sp.exclusive_with):
            continue
        selected.append(sp)
        seen_exclusives.update(sp.exclusive_with or [])
        if len(selected) >= 4: break
    return selected
```

マッチ0でも priority_base 順で上位が選ばれる（フォールバック）。

- **事実ベース原則**: 数字・固有情報は profile.md / 求人シート / 競合調査から引く。`source` 列必須
- **要確認マーカー**: PM ヒアリング待ちの項目は `[要確認]` で明示し、確定まで priority を ? にしておく
- **候補者本人の発話のみ拾う**: 年齢・属性からの憶測で `40代活躍` 等を出さない。候補者が JM で `special_conditions` にチェックしたもののみマッチング対象
- **プライバシー配慮**: 託児所・子育て等は候補者が明示した場合のみ訴求
- **型はめ対象（L0/L1）にも適用可**: ヘッダーは全候補者に出るため、型はめパスでも選定ロジックを通す
