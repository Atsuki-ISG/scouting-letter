# いちご スカウト文 試作サンプル v1

4ブロック分のプロンプト（C, E, A, A'）を使って Opus が生成した試作例。
Gemini/本番APIの出力とは差が出る可能性あり。骨格と構成の妥当性検証が主目的。

---

## サンプル1: ベテラン × α × compact × nursing_veteran

**候補者プロフィール**
- 訪問看護8年、特に終末期対応の経験
- 40代、関東圏在住
- こだわり条件: 高収入
- 自己PR: 「終末期ケアでご家族に寄り添う看護を心がけてきました」

**振り分け判定**
- 優先順5（"高収入" in special_conditions）hit
- skeleton=alpha, tone=compact, attribute=nursing_veteran

**生成結果**
```
🌸【年収500-600万可】月給35-45万／オンコール手当月最大8万

訪問看護8年、特に終末期対応のご実績は、在宅看取り中心の当ステーションで直接活きます。

はじめまして、いちごの柳田です。

8年の訪問看護、特に終末期対応のご実績は、在宅看取り中心のケース構成を持つ当ステーションで直ちに活きます。経験豊富な方の判断力が訪問チーム全体の質を支える場面が多い現場です。2025年5月オープンの富士見台サテライトでは、立ち上げメンバーとして仕組みづくりにも参画できます。

オンコール体制は複数人輪番で月5-6回前後、手当は待機1,000〜6,000円／回、緊急対応で月最大40,000円。月給35-45万、年収500-600万可で、前職からの給与向上も見込めます。

＜募集要項概要＞
...（固定）
```

**文字数**: H 46 + A 47 + B 15 + C 158 + E 100 = 約370字（要項前）

**検証**
- ✅ Gate1a: H で「年収500-600万可／オンコール8万」訴求、高収入志向に刺さる
- ❌ Gate1b: モバイル50字でほぼHで消費、Aは見えない
- ❌ **A と C の重複**: 両方「8年」「終末期」で始まる。プロンプトの重複回避指示を強化する必要
- ✅ 捏造なし: 数値は全て profile.md / job_info から引用
- ✅ compact トーン: 感嘆符なし、断定寄りの文末

**問題の深掘り**: compact ユーザー（高収入志向）は条件判断なので Gate1b 弱くても OK という割り切り想定。ただし A-C 重複は修正必須。

## 3サンプルから見えた課題

### 課題1: A / A' と C の内容重複（確認されたリスク）

3サンプルともに A（または A'）と C の冒頭が候補者経歴の同じ要素（8年・5年・3年）に言及。
v1.1 加圧テストで指摘した問題が実地でも再現。

**対策**:
- A のプロンプトに「経歴の事実に触れる」役割、C のプロンプトに「経歴が何を意味するか・会社でどう活きるか」役割を厳格に分離
- A は「経験年数＋職種の特徴」まで、C は「その経験が当ステーションの○○で直結する」と役割を進める
- プロンプトの `{already_written}` 変数で A の出力を C に渡し、C 側で同じ表現を禁止

**プロンプト修正案の方向**:
- A の出力は C の `{already_written}` として注入
- C プロンプトに「{already_written} の冒頭フレーズと同じ動詞・名詞の組み合わせで始めない」という制約追加

### 課題2: profile.md にない制度を生成している疑い

サンプル3の「事例検討会」「1on1」は profile.md に明記なし（確認済み）。
AI が「訪問看護の標準制度」を推測で埋めている可能性。

**対策**:
- プロンプトの「捏造防止」セクションをさらに厳格化
- 「profile.md に明記されていない制度は書かない」を明示
- 代替として「ハラスメント相談窓口あり」「資格取得全額支援」等の明記ある制度に誘導

### 課題3: compact 版の Gate1b

サンプル1（compact）は想定通り Gate1b 弱い。高収入志向層は条件判断で割り切る前提だが、返信率が低い場合は「compact でも A に候補者情報を入れる」設計に変更する必要あり。

### 課題4: letter トーンでの絵文字運用

サンプル3で H' のみ絵文字使用、本文は絵文字なし。user 確認ポイント:
- letter トーンで絵文字全排除するか、H' のみ許容するか
- letter トーンで 🌸 以外の絵文字（🌿 等）に切り替えるか

---

## 結論

骨格（α/δ）は機能している。4ブロック構成の意図通り Gate 突破が成立することを確認。
ただしプロンプトの役割分離が実装時にさらに厳格化必要。

**次の一手候補**:
1. プロンプトを課題1-2の対策で補正（A と C の役割分離強化、捏造防止強化）
2. 補正版で再度サンプル3本生成して改善確認
3. 実装フェーズに進む（orchestrator, routing, pipeline）
